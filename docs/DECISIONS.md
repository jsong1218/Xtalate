# Xtalate — Engineering Decisions Log

> **Status:** Binding for implementation, subordinate to `MASTER_SPEC.md`. This log records the concrete build-time decisions that the spec leaves to implementation — the ones `docs/ARCHITECTURE_REVIEW.md` §3/§7 flagged as needed before code. Each follows the house style: state the decision, reject at least one alternative. New decisions append here with an ID; a decision that later contradicts the spec triggers a `MASTER_SPEC.md` Revision note in the same change (standing rule, `IMPLEMENTATION_PLAN_v0.1.md` §4).
>
> **Standing rule (added Revision 1.6, from the July 2026 consistency review C5):** any `MASTER_SPEC.md` revision that touches Part 6 (API), Part 7 (Web UI), or Part 9 (Deployment) must **name the affected `IMPLEMENTATION_PLAN_v0.N.md` files in the revision note and edit them in the same change**. Revisions 1.4 and 1.5 did this inconsistently — 1.4 propagated its `LimitsResponse` renames into the v0.5/v0.6 plans but 1.5's `report_retention_days` / history-deletion changes were not carried into the v0.5 M24 and v0.7 M33 deliverables until this review caught the gap. Propagation is part of the revision, not a follow-up.

## D1 — One distribution, `src/` layout (resolves review §4.1, B4)

**Decision.** A single publishable distribution, `xtalate`, built from a `src/xtalate/` package with subpackages `schema`, `sdk`, `parsers`, `exporters`, `capabilities`, `discovery`, `conversion`, `recovery`, `validation`, `cli`. Not eight separately-versioned distributions.

**Rejected — one distribution per component** (`packages/canonical-schema/`, etc., each with its own `pyproject.toml`). Multiplies version numbers, editable-install wiring, and cross-package pinning for a solo maintainer with zero external consumers of a sub-component. The *logical* separation those directories exist for is delivered by the import-graph contract (D5), which works identically on subpackages. Splitting out (e.g. a slim `xtalate-sdk` at v1.0) stays available as an additive packaging change.

**Rejected — flat layout** (`xtalate/` at repo root, no `src/`). The `src/` layout prevents the classic "tests import the un-installed working copy instead of the installed package" footgun and makes the editable install the only import path.

**Naming refinement over `IMPLEMENTATION_PLAN_v0.1.md` M0.** The plan's shorthand listed a combined `formats/`; this log instead keeps `parsers/` and `exporters/` as separate subpackages, because the spec gives them distinct components with distinct "Must NOT" contracts (Part 1 §2) and cites `packages/parsers`/`packages/exporters` throughout. Within each, modules are organized one-per-format (`parsers/xyz.py`, `exporters/xyz.py`). Recorded as `MASTER_SPEC` Revision 1.2 addendum item 8.

## D2 — Python ≥ 3.11

**Decision.** `requires-python = ">=3.11"`.

**Rejected — ≥ 3.9/3.10.** 3.11 gives `tomllib`, faster CPython, and mature `X | None` typing without `__future__` imports; nothing in the target user base (2026 scientific Python) is pinned below it. **Rejected — ≥ 3.12 only.** Needlessly excludes 3.11, still common on HPC/lab images. CI runs 3.11; 3.12 is listed as supported.

**Addendum (M5 CI pass — support both 3.11 and 3.13, and why numpy is *not* pinned to fix stub drift).** The 3.11 floor is **reaffirmed** deliberately: Xtalate is a library + CLI for the computational-chemistry ecosystem (HPC clusters, conda/lab images, institutional Python that lags one to two releases), destined for PyPI. Libraries support a range and let applications pin; the runtime code needs nothing from 3.12+, so dropping 3.11 would foreclose users for zero concrete gain. The 3.11/3.13 CI matrix (`.github/workflows/ci.yml`) stays. Align the floor to SPEC 0 (the scientific-Python support window) rather than to the newest interpreter — revisit only when SPEC 0 ages 3.11 out (~late 2025 onward) or a concrete 3.12+ language feature (e.g. PEP 695 generics in the Plugin SDK) earns its keep, and then bump in one deliberate change.

The M5 work surfaced a **numpy type-stub drift**: numpy **2.5 requires Python ≥ 3.12**, so the 3.13 leg resolves numpy 2.5 while the 3.11 leg tops out at numpy 2.4.6, and the two stub generations type the same expression differently (`arr[intp_array]` is `ndarray` under 2.5's stubs, `Any` under 2.4's) — which failed `mypy --strict`'s `no-any-return` on the 3.11 leg only. **We do not fix this with a numpy version pin**, because no pin can: a `numpy>=2.5` *lower* bound is uninstallable on 3.11 (there is no numpy 2.5 for it), and a `numpy<2.5` *upper* cap — though it would force both legs onto the same 2.4.x stubs — is exactly the foundational-dependency upper-cap that breaks downstream resolution for anyone installing Xtalate beside newer numpy, and it merely defers the same split to the next numpy release that drops a Python. The runtime `numpy>=1.26` (D4) is therefore left uncapped.

The correct guard is **process, and it already exists**: the CI 3.11 leg runs `mypy` against the older (stricter) numpy stubs and *did* catch the drift; the only gap was the maintainer's local loop, which had run `mypy` against only the 3.13/numpy-2.5 resolution. Standing rule for numpy-typed code: type-check against the **oldest supported numpy** (the 3.11 resolution) before pushing — CI is the backstop, not the first line. The fix itself is written to satisfy *both* stub generations (bind the indexed result to an `ndarray`-annotated local instead of returning the bare index expression), which is the durable pattern for numpy-return typing across stub versions. Recorded so the next stub-drift failure is diagnosed as an expected consequence of the honest two-numpy matrix, not re-litigated as a reason to drop a Python or cap numpy.

**Second addendum (scheduled floor review, July 2026 — reaffirm 3.11).** The first addendum set a review trigger — "revisit only when SPEC 0 ages 3.11 out (~late 2025 onward)…" — and that date has now passed, so per the D-log's own discipline the scheduled review is performed here rather than allowed to lapse silently. **Outcome: the ≥ 3.11 floor is reaffirmed, unchanged.** The reasoning still holds: Xtalate is a PyPI library + CLI for the computational-chemistry ecosystem (HPC clusters, conda/lab images, institutional Python that lags one to two releases); a library supports a range and lets applications pin, the runtime needs nothing from 3.12+, and SPEC 0's support window still includes 3.11 as of this review. No concrete 3.12+ language feature (e.g. PEP 695 generics in the Plugin SDK) has yet earned its keep. This is a dated reaffirmation, not a bump. The next natural review slot is the v0.2 **M11** release pass, the next milestone that touches `pyproject.toml`; if a bump is warranted then it lands there in one deliberate change.

## D3 — Build backend: Hatchling

**Decision.** `hatchling>=1.27` (PEP 639 SPDX `license`/`license-files` support).

**Rejected — setuptools.** Works, but its `src/` + PEP 639 configuration is heavier and its defaults leak (accidental package discovery). **Rejected — Flit.** Elegant for pure-Python single modules, but less ergonomic once console scripts, data files (golden corpus), and future native bits appear. Hatchling is the modern middle with minimal config.

## D4 — Dependencies

**Decision.** Core runtime: `pydantic>=2.7`, `numpy>=1.26`. Sole scientific-I/O dependency: **ASE** (added when the extXYZ parser lands, M3c — see D7), version-pinned at that point. Dev extra: `pytest`, `ruff`, `mypy`, `import-linter`.

**Rejected — adding ASE now.** It is a heavy transitive tree; nothing before M3c imports it, and keeping the M0–M3b dependency surface to pydantic+numpy keeps installs fast and the laundering-test boundary explicit. **Rejected — pymatgen as a core dep.** D7 hand-rolls POSCAR/CONTCAR, so pymatgen is not needed in v0.1 at all.

## D5 — Tooling and P2 enforcement

**Decision.** `ruff` (lint + format), `mypy --strict`, `import-linter`, `pytest` — all four run on every PR (`.github/workflows/ci.yml`) from day one, not as stubs. The import-linter `layers` contract (in `pyproject.toml`) encodes the acyclic graph of Part 1 §5.1 and is the mechanical enforcement of **P2**.

**Rejected — a hand-written import-check script / deferred "lint stub"** (as the roadmap's week 1 suggested). A real `import-linter` contract is barely more effort than a stub and actually fails CI on a violation; a stub that passes vacuously trains contributors to ignore it. **Rejected — mypy non-strict.** For a library whose product is correctness, strict typing from an empty tree costs nothing and prevents a slow slide.

## D6 — CLI framework: argparse (stdlib)

**Decision.** The `xtalate` CLI (Appendix A) is built on the standard library's `argparse`, with `xtalate.cli:main` as the console-script entry point.

**Rejected — Typer / Click.** Nicer help ergonomics, but each is a runtime dependency for a four-subcommand surface (`inspect`/`convert`/`validate`/`capabilities`) that argparse covers, and the CLI is a thin JSON presenter (Appendix A) where dependency minimalism matters more than decorator sugar. Revisit only if the surface grows substantially post-v0.1.

## D7 — Per-format implementation strategy (resolves review B5)

**Decision.** Hand-rolled **XYZ** (trivial grammar; avoids library-default laundering entirely). ASE-backed **extXYZ** with the mandatory default-laundering suite (Part 3 §2) — the `Properties=`/`Lattice=` grammar is where battle-tested code earns its keep. Hand-rolled **POSCAR/CONTCAR** (well-documented; hand-rolling gives full control of the selective-dynamics → `Constraint` mapping and avoids a pymatgen dependency). Net effect: **ASE is the only scientific dependency in v0.1**, and it backs exactly one format.

**Rejected — ASE/pymatgen behind all four formats.** Maximizes reuse but also maximizes the laundering-test burden (every library-invented default — zero cell, `pbc=(F,F,F)`, zero momenta — must be caught and nulled, Part 3 §2) and drags two heavy deps in for formats a few hundred lines of hand-rolled code handle deterministically.

## D8 — Array serialization and golden-file equality (resolves review B6)

**Decision.** NumPy arrays (float64 in memory) serialize to nested JSON lists using Python's default shortest-round-tripping float `repr` (what `json.dumps` emits). Golden-file comparison **deserializes both sides and compares arrays with exact `==`** (integers/counts) or `np.array_equal` on the parsed values — never string-compares JSON text. This makes float64 → JSON → float64 lossless and keeps the golden corpus deterministic across platforms.

**Rejected — fixed-decimal string formatting** (e.g. `%.12f`). Truncates float64 and reintroduces exactly the representational ambiguity the tolerance model (Part 5 §4) exists to reason about explicitly. **Rejected — comparing serialized text.** Whitespace/key-order/float-format differences produce false failures; equality is a property of the *values*, not their serialization.

## D11 — Development environment: project-local `.venv` (stdlib), not conda

**Decision.** Development uses a project-local virtual environment at `.venv/` created with the standard library (`python -m venv .venv`), then `pip install -e ".[dev]"`. `.venv/` is already gitignored and is never committed. This matches how CI provisions its environment (pip + `actions/setup-python`) and how Persona 2 consumes the project (`pip install xtalate` from PyPI, Part 9 §3).

**Rejected — conda.** The maintainer uses conda for other projects but deliberately not this one. The usual computational-chemistry reason to reach for conda is painful compiled dependencies (spglib, RDKit, MKL); Xtalate's dependency surface is pydantic + numpy + (later) ASE — all pure-Python with reliable pip wheels, and pymatgen was rejected (D7) — so conda buys nothing here while adding environment-resolution overhead and a dev/CI toolchain split (CI is pip-based). A project-local `.venv` also shadows an active conda `base` cleanly, so the maintainer need not `conda deactivate` to work on this repo.

**Rejected (for now) — uv.** `uv` is a fine faster alternative and stays entirely in the pip/PyPI world, but it is not currently installed on the machine (adopting it means a global install first), and the speed benefit is marginal for a three-package dependency surface. `uv venv && uv pip install -e ".[dev]"` remains a drop-in swap later — it reads the same `pyproject.toml` unchanged — if install/resolve time ever becomes a friction point.

**Python version.** The `.venv` is built from the machine's available interpreter, **Python 3.13** (conda `base`); no 3.11 is installed locally, and installing one via conda would contradict this decision. Since CI pins the supported floor (3.11), the `.venv`'s 3.13 would otherwise be untested — so CI was changed to a matrix over **3.11 (floor) and 3.13 (dev)** (`.github/workflows/ci.yml`), and 3.13 was added to the `pyproject.toml` classifiers (it was already permitted by `requires-python = ">=3.11"`). Dev and CI therefore share both endpoints; the M0 gate suite passed identically on 3.13 locally and is exercised on 3.11 in CI.

## D12 — `custom_per_atom` / `custom_per_frame` accept JSON-scalar lists, not only numeric arrays (M1)

**Decision.** A `UserMetadata.custom_per_atom` / `custom_per_frame` value is a sequence whose first dimension is N / F and is **either** a numeric `float64` ndarray (extXYZ extra columns) **or** a length-N / length-F `list` of JSON scalars (per-frame/per-atom free text, e.g. XYZ comment lines). Implemented as a `left_to_right` union (`ArrayNx | list[JsonValue]`) so numeric input coerces to an ndarray and only non-numeric input (strings) falls through to the list form. The first-dimension check (== N / F) runs on both branches. Recorded as `MASTER_SPEC` Part 2 §3.10 Revision 1.3.

**Context.** `MASTER_SPEC` §3.10 originally typed both fields as `Array[(N/F, ...)]` (numeric), but the worked example §8.1 and the carry-through routing rule §6.1 both put XYZ per-frame **string** comments in `custom_per_frame["xyz:comment"]` — content the numeric-only type could not hold. The M1 golden fixture makes this executable, forcing the reconciliation now rather than at first extXYZ/XYZ parse.

**Rejected — keep the array-only type and route XYZ comments elsewhere** (e.g. `custom_global`). It would contradict §6.1's explicit "per-frame free text → `custom_per_frame`" routing and §8.1's worked object, and would lose the per-frame association (which comment belongs to which frame) that a length-F list preserves. **Rejected — a dedicated `custom_per_frame_text` field.** A parallel string-only surface doubles the carry-through routing table and the Capability-Matrix "arbitrary arrays" row for no gain; one first-dim-N/F surface that is agnostic to element type (numeric vs JSON scalar) is simpler and still shape-validated.

## D13 — POSCAR exporter writes **Cartesian** coordinates, not fractional (M3b)

**Decision.** The POSCAR/CONTCAR exporter emits coordinates in `Cartesian` mode with a scaling factor of `1.0` (the canonical `cell.lattice_vectors` are already absolute Å, Part 2 §3.4), even though the format natively supports `Direct` (fractional) too. Canonical positions are Cartesian by definition (Part 2 §4), so Cartesian output is written verbatim — no matrix inversion — which makes a parse→export→parse identity round-trip reproduce positions *exactly* (`np.array_equal`), not merely within tolerance.

**Rejected — write `Direct` (fractional) to mirror VASP's common convention.** Fractional output requires `frac = cart · lattice⁻¹`, and `cart · lattice⁻¹ · lattice` is only equal to `cart` up to floating-point rounding for a general (non-diagonal) lattice. That reintroduces representational error into the one place the strict-profile round-trip should be exact, and would force the identity test onto tolerance comparison rather than equality. VASP reads both modes, so nothing downstream is lost; the source's original mode is still recorded in `provenance.original_coordinate_system`. A future exporter option can offer `Direct` output once the tolerance model (Part 5 §4) governs that path.

## D14 — Identity round-trip equality is over **scientific content**, excluding `provenance` (M3a)

**Decision.** The identity round-trip criterion (`A → Canonical → A' → Canonical'`, Part 3 §3) compares the two objects' *scientific content* — `frames`, `trajectory`, `simulation`, `user_metadata` — for exact equality (deserialize-then-compare, D8), and deliberately **excludes `provenance`**. Golden-file comparison (parse vs hand-verified `expected.canonical.json`) compares everything *except* the volatile parse-event bookkeeping in `provenance.history` (wall-clock `timestamp`, `tool_version`, `parser_version`), which are normalised on both sides.

**Rejected — whole-object equality including `provenance`.** `provenance` records *how this particular file was read*: its `source_filename`, and the `original_coordinate_system` it happened to encode. A faithful re-export legitimately changes these — e.g. a POSCAR parsed from `Direct` source re-exports as `Cartesian` (D13), flipping `original_coordinate_system` from `"fractional"` to `"cartesian"` with zero information loss. Demanding provenance equality would make a correct round-trip fail on metadata that is *supposed* to reflect the differing inputs, and would force every parser to fabricate identical timestamps. Excluding provenance from the *equality* check does not weaken it: provenance is independently asserted by the golden tests, where its non-volatile fields must match exactly.

## D15 — One POSCAR implementation, registered twice as `poscar` and `contcar` (M3b)

**Decision.** POSCAR and CONTCAR are byte-identical formats (Part 3 §6.1), so a single `PoscarParser` / `PoscarExporter` class backs both, instantiated twice under two `format_id`s (`poscar`, `contcar`) that differ only in (a) the conventional filename each sniffs at confidence 1.0 and (b) a small sniff bias — POSCAR wins a nameless structural tie (`base_score` 0.6 vs 0.55), while a velocity/predictor-corrector tail tips the result to CONTCAR (`tail_bonus`), all surfaced as an `ambiguous` sniff rather than a forced winner (Part 3 §6.1). Factory helpers `make_poscar_parser()` / `make_contcar_parser()` (and the exporter pair) produce the two configured instances.

**Rejected — two independent parser/exporter classes.** Duplicates the entire POSCAR grammar (scaling factor, lattice, VASP-4 detection, selective dynamics, coordinate conversion, velocity tail) for two readings whose canonical output is identical by construction — the maintenance and drift cost the §6.1 "same canonical fields" guarantee exists to avoid. The one-class/two-registrations shape keeps the scientific logic single-sourced and localises the difference to sniff bias and the reported `source_format` label.

## D16 — Additive stable `ParseIssue` codes for the XYZ and POSCAR error contracts (M3a/M3b)

**Decision.** The two hand-rolled parsers emit stable machine `code`s beyond the single `XYZ_INCONSISTENT_ATOM_COUNT` the spec names explicitly (Part 3 §5 rule 4). XYZ: `XYZ_INCONSISTENT_ATOM_COUNT` (with `recovery_hint="truncate_at_last_valid_frame"`), `XYZ_EMPTY`, `XYZ_MALFORMED_HEADER`, `XYZ_INVALID_SYMBOL`, `XYZ_MALFORMED_COORDINATE`. POSCAR/CONTCAR: `POSCAR_MISSING_SPECIES` (with `recovery_hint="supply_species"`, the VASP-4 case of Part 3 §3 n.1), `POSCAR_INCONSISTENT_ATOM_COUNT`, `POSCAR_MALFORMED`, `POSCAR_INVALID_SYMBOL`, and the warning `POSCAR_PREDICTOR_CORRECTOR_CARRIED` (Part 3 §3 n.12). All obey the §5 contract: an error precludes a Canonical Object (raised as `ParseError`); a warning accompanies success (returned in `ParseResult.issues`); nothing is skipped silently (§5 rule 2). Recorded as `MASTER_SPEC` Revision 1.2 addendum item 12.

**Rejected — a single generic `PARSE_ERROR` code per format.** Collapsing distinct failures (missing species vs truncated file vs bad symbol) into one code destroys the machine-actionable distinction the error contract exists for — the Recovery Engine (Part 4 §3) branches on `recovery_hint`, and a `missing species` recoverable error must be distinguishable from an unrecoverable malformation. The spec's own worked example (§5 rule 4) establishes the granular, format-prefixed naming convention these codes follow.

## D17 — ASE dependency: pinned `>=3.29,<4`, typed as an untyped boundary (M3c)

**Decision.** extXYZ lands the sole scientific-I/O dependency (D4, D7): `ase>=3.29,<4` in the core `dependencies` (not the `dev` extra — it is needed at runtime by the extXYZ parser/exporter). ASE ships a `py.typed` marker but is largely unannotated, so under `mypy --strict` every ASE call site would trip `no-untyped-call`. Rather than scatter `# type: ignore`s, ASE is treated as an untyped I/O workhorse confined to the two `extxyz` wrapper modules via a single `pyproject.toml` mypy override (`module = "ase.*"`, `follow_imports = "skip"`), which makes ASE symbols `Any`. Its behavior is pinned by the laundering + golden tests, not by its type stubs.

**Rejected — a tighter pin (`==3.29.*`).** ASE's minor releases have historically changed I/O defaults, and this package's correctness depends on ASE's exact laundering surface (zero cell, `pbc`, momenta) — an argument *for* pinning tightly. But the golden corpus and the default-laundering suite (Part 8 §1.1) already fail CI loudly on any behavioral drift, so a hard equality pin would trade a real maintenance cost (blocking every ASE patch release) for protection the tests already provide. The `<4` cap guards against a major-version break; the floor is the version M3c was developed and tested against. **Rejected — vendoring only the extxyz reader.** Copying ASE's `Properties=`/`Lattice=` parser into the tree removes the dependency but forks battle-tested code that the community keeps correct (D7's entire rationale for choosing ASE for this one format), and would still need the same laundering wrapper. **Rejected — keeping ASE in the `dev` extra.** It is imported by shipped runtime code (`xtalate.parsers.extxyz`), so a `pip install xtalate` without it would `ImportError` the moment an extXYZ file is sniffed — it is a hard runtime dependency, not a test-only one.

## D18 — extXYZ field mapping: map the unit-/sign-safe fields, carry the rest verbatim (M3c)

**Decision.** The ASE-backed extXYZ parser maps a source value into a canonical *field* only when the mapping is unambiguous in both unit and sign; otherwise it carries the value through verbatim under a namespaced `user_metadata` key, so nothing is dropped and nothing is fabricated (P1, P3). Concretely:

- **Mapped (unit-/sign-safe):** `positions` (Å), `Lattice=`→`cell.lattice_vectors` (Å), `masses`→`atoms.masses` (u), `energy`→`electronic.total_energy` (eV), `forces`→`dynamics.forces` (eV/Å), a per-atom `charge`/`charges` column→`electronic.charges` (e, cation-positive), a per-atom `magmoms` column→`electronic.magnetic_moments` (μB, spin-up-positive), and a `momenta` column→`dynamics.velocities`, converted from ASE's internal velocity unit to canonical Å/fs (factor `ase.units.fs`). All match the canonical unit/sign conventions of Part 2 §3.1/§3.7.1 exactly.
- **Laundered to absence:** ASE always returns a fully-populated `Atoms`, so an all-zero cell (no `Lattice=`) → `cell = None`; synthesised zero momenta (no `momenta` column) → `velocities = None`; computed-from-Z masses (no `masses` column) → `masses = None`. A `Lattice=` with no `pbc=` key keeps the (real) cell and takes `pbc=(T,T,T)` by extXYZ convention, recorded in `parse_notes` (the analogue of POSCAR's format-defined-PBC note, Part 3 §3 n.3) — never presented as declared source data.
- **Carried verbatim (no unit-/sign-safe home):** `stress`/`virial`→`custom_per_frame["extxyz:stress"]`; arbitrary `Properties=` columns→`custom_per_atom["extxyz:<name>"]`; comment key=value pairs→`custom_per_frame["extxyz:<key>"]`; and any *other* calculator result ASE produces→`custom_per_frame["extxyz:<key>"]` with an `EXTXYZ_UNMAPPED_RESULT_CARRIED` warning.

This is a deliberate, recorded **deviation from the read-side capability declaration of `MASTER_SPEC` Part 3 §4.2**, which lists `electronic.stress` as `partial`. In v0.1 the extXYZ parser does *not* populate `electronic.stress`; it carries stress verbatim instead, so `electronic.stress` is declared `none` (absent from the parser's `fields`) and a `lossy_notes` entry states where stress lives. Recorded as `MASTER_SPEC` Revision 1.2 addendum item 14.

**Rejected — map `stress` into `electronic.stress` anyway.** ASE surfaces stress as a Voigt-6 tensor in eV/Å³, but its *sign convention* (compression- vs tension-positive) cannot be reconciled with the canonical tension-positive convention (Part 2 §3.7.1) without a convention the file does not declare — mapping it would risk a silent sign flip, which is a worse failure than honest carry-through (a wrong number presented as data vs a correct number presented as an un-interpreted extra). Carrying it verbatim loses nothing (the values are preserved and reported) and fabricates nothing. When a later version adds a source-of-convention signal (or a fixture pins ASE's sign), stress can be promoted to `electronic.stress` additively. **Rejected — carry *everything* (velocities, charges, energy too) as custom arrays** to sidestep all unit reasoning. That would abdicate the parser's core job — populating the canonical fields the Capability Matrix reasons over (P5) — for the fields whose units *are* unambiguous, degrading loss prediction and the whole conversion pipeline for no correctness gain. **Rejected — drop the unmappable extras silently.** The exact silent loss the project exists to prevent (P1); every carried value is namespaced, shape-validated, and reported.

## D19 — Pre-flight `PARTIAL` handling: predict Preserved, surface the condition as a Warning (M4)

**Decision.** In the pre-flight diff (Part 3 §4.3), a source-present path whose target write-capability is `PARTIAL` is predicted **Preserved**, with the declared `FieldCapability.notes` condition surfaced *both* as the preserved entry's `detail` *and* as a `capability`-source `ReportWarning` (`code="PARTIAL_CAPABILITY"`). The per-object condition is **not evaluated** in v0.1 — e.g. POSCAR's `cell.pbc` PARTIAL ("only fully periodic") is reported preserved-with-warning whether or not `pbc == (T,T,T)`. The spec's own `lossy_notes` become `FORMAT_LOSSY_NOTE` warnings the same way (§4.3 rule 5).

**Rationale.** §4.3 says PARTIAL predicts "Preserved **or** Removed depending on the declared condition." The conditions are prose (`notes`), not machine-evaluable in general, and building a per-condition evaluator is disproportionate for v0.1's handful of PARTIAL cases. Defaulting to Preserved-**with-a-prominent-warning** keeps the philosophy intact: the caveat is always shown (never silent, P1), and the Validation Engine (M5) re-parses the output and diffs it, so a PARTIAL field the exporter could not in fact represent is *caught after the fact* — the report is honest about the caveat, and validation is the backstop that turns an unmet condition into a flagged discrepancy rather than a silent loss.

**Rejected — predict Removed for every PARTIAL.** Over-reports loss: it would tell the user a field they will actually keep (POSCAR pbc for the overwhelmingly common fully-periodic case) is being dropped, training them to ignore the report. **Rejected — evaluate each condition now.** A prose-condition evaluator (or a per-field predicate registry) is real machinery for a case validation already covers; deferred until the PARTIAL set is large enough to justify it (P6).

## D20 — `write_plan` is materialized as a filtered Canonical Object, not a side-channel argument (M4)

**Decision.** The engine enforces the `write_plan` (Part 4 §1) by constructing `canonical′` — a copy of the source with every non-plan field set to `None` — and handing *that* to the ordinary `ExporterPlugin.export(canonical, stream)`. The exporter interface is **unchanged** from its M2 frozen shape; there is no `write_plan` parameter. Because exporters honor the absence convention (rule 2: "never fabricates values for absent fields"), an exporter given `canonical′` writes exactly the plan — no more, no less — structurally rather than by trust. `canonical′` is returned on `ConversionResult.canonical_out` and is precisely the *expected object* the Validation Engine will diff the re-parsed output against (Part 5 §1).

**Rejected — add a `write_plan: set[str]` parameter to `export()`** (matching the sequence diagram's `export(canonical′, write_plan)` literally). It changes the SDK ABC every existing and third-party exporter implements, for information the filtered object already carries: a field excluded from the plan is `None` on `canonical′`, which an absence-honoring exporter must already skip. The two would then have to be kept consistent (a plan that says "drop forces" plus an object that still has forces is a contradiction the exporter would have to arbitrate). Materializing the plan as the object collapses plan and data into one source of truth. **Rejected — export the unfiltered object and rely on capability declarations.** Output bytes would be identical for today's disciplined exporters, but `canonical_out` would then *not* equal the expected object (it would still carry dropped fields), breaking the M5 validation reference and losing the structural guarantee that a less-disciplined future exporter cannot leak an excluded field.

## D21 — `atoms.atomic_numbers` is excluded from the diff and the completeness invariant (M4)

**Decision.** The pre-flight diff and the completeness-invariant assertion skip `atoms.atomic_numbers`. It is a *derived mirror* of `atoms.symbols` (Part 2 §3.3, "derived from symbols at construction"), always present, and reconstituted by any format that writes symbols — so counting it as an independently losable field would force every target to "preserve" it and would clutter every report with a non-fact.

**Rejected — treat it as a normal field.** Since no exporter declares `atoms.atomic_numbers` writable, the diff would mark it **Removed** on every single conversion — a phantom loss entry that is never a real loss (the numbers are recoverable from the symbols that *are* written), undermining the report's signal-to-noise exactly where trust is built. **Rejected — special-case it as always-Preserved.** Also honest, but it still adds a noise entry to every report for a field that is not independent information; excluding it (like `provenance`, already excluded from presence, Part 2 §3.11) is the cleaner statement that it is not part of the source's discoverable content.

## D22 — Recovery is preset-only in v0.1 (M5)

**Decision.** The Recovery Engine (Part 4 §3) resolves scenarios from a caller-supplied `recovery_choices` dict — one preset per scenario, `{"choice": str, "parameters": {...}}` — and nothing else. There is no interactive prompt, no `awaiting_recovery`/resume state machine, and no per-scenario default: a scenario with no matching preset (or a preset for a scenario v0.1 cannot resolve) refuses the *whole* conversion (`status="refused"`, `refusal.code="RECOVERY_REQUIRED"`). Resolution is all-or-nothing — a single unresolved scenario yields `RecoveryResult.canonical is None`, never a partially-recovered object. Every applied preset still records an `Assumption` with `origin="preset"`, and (for fabricative scenarios) a `SuppliedEntry`, exactly as an interactive choice would.

**Rationale.** The interactive Recovery Workflow and the `awaiting_recovery` job state belong to the v0.5 Service layer (Part 6 §3); v0.1 is a pure library + CLI, where the caller *is* the decision-maker and supplies choices up front (the CLI's repeatable `--recover SCENARIO=CHOICE[,param=value…]`, M6). Preset-only keeps the philosophy exact — a choice is still explicit, recorded, and never invented (**P4**) — while deferring the stateful machinery to the version that has a UI to drive it. All-or-nothing refusal is what prevents the "half-recovered structure presented as complete" failure, which would be the very silent loss the engine exists to prevent.

**Rejected — a per-scenario default (e.g. always `bounding_box`, `padding=0`).** A fabricated lattice nobody chose is exactly the silent fabrication **P4** forbids; the padding value alone changes the physics (periodic contacts). **Rejected — resolve the scenarios it can and refuse only the rest, returning a partial object.** A structure missing a target-required field but presented as converted is silent loss; refusal must be atomic. **Rejected — build the `awaiting_recovery` resume state now.** Real stateful machinery (persisted paused jobs, expiry→refusal) for a library with no server; deferred to v0.5 at the seam Part 6 §3 already defines (**P6**).

## D23 — Exporters declare their reorder via an additive `atom_permutation()` seam (M5)

**Decision.** `ExporterPlugin` gains a **concrete, non-abstract** method `atom_permutation(canonical) -> list[int] | None`, defaulting to `None` (no reorder / identity). An exporter that reorders atoms on write overrides it to return the permutation `perm` where output position *i* holds source atom `perm[i]`; POSCAR (which groups atoms by element) derives its map from the *same* `_grouping()` helper its `export()` uses, so the written order and the declared map can never disagree. The `positions_rmsd` and `species_preservation` checks (Part 5 §2) apply this map before comparing.

**Rationale.** Part 5 §2 requires that "exporters that reorder atoms must emit an index permutation map alongside the output." The method is added as a concrete default rather than an abstract method so the frozen `export`/`capabilities` contract (M2) and every existing/third-party exporter keep working unchanged — an exporter that preserves source order needs no edit, and identity is the correct answer for it. Sharing one grouping helper between `export` and `atom_permutation` is what makes the map *trustworthy*: a bug in the grouping breaks both symmetrically and is caught by the golden round-trips, rather than the map silently disagreeing with the bytes.

**Rejected — an abstract `atom_permutation` every exporter must implement.** Breaks the frozen SDK ABC and forces boilerplate `return None` on formats that never reorder. **Rejected — return the map from `export()` as a second value / out-parameter.** Changes the frozen `export(canonical, stream) -> None` signature and couples the map's lifetime to a write call; a separate query the Validation Engine calls independently is cleaner. **Rejected — have validation re-derive the grouping itself.** Duplicates the exporter's ordering logic in a second place that could drift from it — the exact asymmetry that would let a reorder bug pass validation.

## D24 — `numeric_precision` added additively to `FormatCapabilities`; v0.1 exporters declare full precision (M5)

**Decision.** `FormatCapabilities` gains an optional `numeric_precision: dict[str, int | None]` map (canonical path → declared decimal count, or `None`/absent = full precision), the machine-readable generalization of `lossy_notes` that Part 5 §4.2 requires for the representational-bound tolerance. The v0.1 POSCAR exporter declares **no** entries: it writes Cartesian coordinates with `repr(float)` (D13), which round-trips exactly, so every field's representational bound is `0.0` and the base tolerances of §4.3 govern unchanged.

**Rationale.** §4.2 (flagged in §8 as an additive `FormatCapabilities` extension) scales tolerances by declared precision so low-precision formats (CIF's 4-decimal fractional coordinates) are not false-failed and high-precision formats are not given a free pass. The field is added now so the mechanism and its report plumbing (`tolerance_applied.representational_bound_ang`) exist and are tested, even though the only v0.1 exporter is full-precision. The fractional-format lattice-scaling branch (`× max‖Lᵢ‖`) is implemented as the formula but unexercised until a fractional exporter (CIF, v0.2+) lands — the seam is declared, not speculatively built out (**P6**).

**Rejected — hard-code a per-format precision table inside the Validation Engine.** Puts format knowledge in the wrong layer; capabilities are the one place a format declares its own limits, and a third-party exporter must be able to declare precision without patching validation. **Rejected — defer the field entirely until a fractional format needs it.** Then the tolerance machinery would ship untested against the mechanism it exists for, and adding it later would be a capability-schema change mid-stream rather than an already-present default.

## D25 — `atoms.masses` tolerance base, absent from the §4.3 table (M5)

**Decision.** `numeric_field_fidelity` (Part 5 §2) validates `atoms.masses`, but the §4.3 default-tolerance table lists no masses row. It is given the forces/velocities base (`warn 1×10⁻⁶`, `fail 1×10⁻⁴`, in u) as the nearest-precision analogue.

**Rationale.** The check catalog explicitly names `atoms.masses` among the eight numeric fields, so a tolerance is required to run it; the table simply omits it because no v0.1 conversion writes masses through a validated path (POSCAR carries no masses; XYZ writes only symbols+positions). A concrete, documented base lets the check execute honestly if a future format does write masses, rather than crashing on a missing key or silently skipping. Isotopic mass precision is comparable to the force/velocity decimal precision these bases target. Per the M5 cut-line, `numeric_field_fidelity` breadth for fields no v0.1 format writes is explicitly the trimmable part — this records the chosen value rather than leaving it implicit.

**Rejected — omit masses from the check.** Contradicts the §2 catalog, which names it. **Rejected — invent a bespoke masses row without saying so.** The value must be recorded with a rationale like every other tolerance; an undocumented threshold is the kind of silent judgment the log exists to surface.

## D26 — Validation reads the Conversion Report through a structural `ConversionReportView` Protocol (M5)

**Decision.** The Validation Engine sits *below* `conversion` in the import graph (Part 1 §5.1), so it may not import `conversion.report.ConversionReport`. `ValidationEngine.validate` accepts the report as a `ConversionReportView` — a `typing.Protocol` declared in `validation.engine` whose read-only property members (`report_id`, `preserved`, `removed`, `supplied`, `assumptions`) the concrete `ConversionReport` satisfies structurally, with no import. The Conversion Engine (top layer) passes its own report object in.

**Rationale.** This keeps the P2/layering contract (enforced by import-linter, D5) intact while letting validation consume the actual report object rather than a hand-copied bundle of primitives. Structural typing is exactly the tool for "depend on a shape without depending on the module that defines it." Members are read-only *properties* (not bare attributes) so the collection types are covariant — `list[PreservedEntry]` satisfies `Sequence[_Entry]`, which a settable Protocol attribute would reject as invariant.

**Rejected — pass the report's fields as loose primitives** (`preserved_paths: set[str]`, `supplied: list[tuple]`, …). Loses the typed report as a single source of truth and invites drift between what the engine assembles and what validation re-derives. **Rejected — move the report schema down into a shared lower layer** so both can import it. A larger refactor of a frozen M4 schema's home for one consumer; the Protocol is the minimal, reversible seam. **Rejected — annotate the parameter `Any`.** Discards the type checking that catches a genuine shape mismatch at the call site.

## D27 — `absence_conformance` and `report_consistency` check scoping in v0.1 (M5)

**Decision.** Two checks are scoped more precisely than the Part 5 §6 worked-example prose:

- **`absence_conformance`** asserts absence only for removed paths that are **not also** in `preserved`. The frame-reduction `removed` entry (`atoms.positions`, "9 of 10 frames dropped") names a path that is genuinely present for the *retained* frame; asserting it absent would false-fail. That reduction is verified by `frame_count` instead. (The §6 fixture cosmetically lists `atoms.positions` under `absence_conformance` with a frames-0–8 note; the meaningful check is the frame count, so v0.1 reports the frame-drop there and the count under `frame_count`.)
- **`report_consistency`** independently re-verifies part (a) it can see from the final report alone — every `supplied.from_assumption` resolves to a recorded `Assumption` — and reports the completeness invariant as satisfied (the Conversion Engine has already asserted it unconditionally at finalization, `CompletenessInvariantError`, before validation runs). It does **not** reconstruct the pre-flight draft to diff preflight-vs-final in v0.1; `preflight_final_deltas` is reported as the count of recorded Assumptions (each divergence traces to one).

**Rationale.** Both keep the checks *honest* rather than performing the fixture's exact bookkeeping: a preserved-and-removed path is not a real absence violation, and the completeness invariant is already a hard runtime assertion upstream, so validation re-affirms the part that is independently checkable (supplied traceability) without duplicating machinery. Full preflight-vs-final structural diffing needs the engine to thread the draft report into `convert()`; that is deferred (the M5 cut-line explicitly allows trimming report `detail` richness, never the invariant checks themselves — supplied traceability is kept).

**Rejected — assert every `removed` path absent literally.** False-fails on the frame-drop path, which is present for the retained frame. **Rejected — carry the pre-flight draft into `convert()` now to diff it fully.** Extra plumbing for a divergence already bounded by the recorded Assumptions (which validation does check); the seam stays open for v0.2. **Rejected — skip `report_consistency` because the engine already asserts completeness.** Validation's job is independent re-checking; re-verifying supplied-traceability from the report is cheap defense-in-depth, and the check's presence (not omission) is itself part of the §3 "report every check" discipline.

## D28 — Discovery Report `fields` is a fixed 16-path leaf list; metadata goes to `structure`/`extras`; `default_registry()` placement (M6)

**Decision.** The `DiscoveryReport.fields` inventory is emitted over a **fixed, ordered list of 16 canonical scientific leaf paths** (Part 2 §3.3–§3.7 plus `frame.time` and `trajectory.timestep`), in the Part 3 §6.3 worked-example order — *not* over whatever `field_presence()` happens to return. `atoms.atomic_numbers` (a derived mirror) is excluded; root metadata containers (`simulation.*`, `user_metadata.tags/annotations`) are summarized under `structure`, and the per-file `custom_*` / `simulation.extra` keys are listed under `extras`. The standard registry is built by a top-level `xtalate.registry.default_registry()` factory (all built-in parsers + exporters), not a method on `capabilities.Registry`.

**Rationale.** §6.2 requires the `fields` list be *complete over the canonical leaf paths* so "not shown" can never be mistaken for "not checked" — that completeness is only guaranteed by iterating a fixed schedule of paths and looking each up, rather than reporting the presence map's dynamic entries (which include derived and per-file-varying keys). Metadata containers are format-specific and unbounded, so they belong in the free-form `structure`/`extras`, exactly as the spec's example places them. `default_registry()` sits at the package root because the registry *machinery* must not import concrete parsers/exporters (that inverts the P2 import graph, D5); the wiring that composes them lives above both — the same reason the CLI, not the engine, assembles the pipeline.

**Rejected — derive `fields` directly from `field_presence()`.** It would surface `atoms.atomic_numbers` and the dynamic `custom_*` keys as if they were losable leaf fields, and could omit an absent path entirely (presence lists present/mixed paths naturally) — breaking the completeness guarantee. **Rejected — put `default_registry()` on `capabilities.Registry` or in `capabilities/__init__`.** That module would then depend on `parsers`/`exporters`, inverting the layered import contract; the factory is deliberately a thin composition root instead.

## D29 — Offline full re-parse re-validation refuses when the conversion fabricated fields (M6)

**Decision.** The CLI's `validate --output --source --conversion-report` (offline full re-parse re-validation, Part 5 §4.5 point 2) reconstructs the expected object from the source file filtered through the report's `preserved` write plan. When the Conversion Report has any `supplied` (recovery-fabricated) entries, it **refuses with a usage error** rather than validate, and points the user to re-thresholding the original Validation Report instead. Re-thresholding (`validate --validation-report` alone) and full re-parse of non-fabricative conversions both work normally.

**Rationale.** A fabricated value (e.g. a `bounding_box` lattice) exists in neither the source file nor the Conversion Report's path lists (which record *which* fields, not their values), so it cannot be reconstructed offline. Validating against an expected object that is silently missing the fabricated field would produce a *wrong* result that looks authoritative — precisely the silent failure the project exists to prevent (**P1**). Refusing loudly, with a pointer to the operation that does work (re-thresholding needs only the stored measurements), is the honest v0.1 behavior. The spec's §4.5 claim that the expected object is reconstructable "from the source file and the report's path lists" holds for non-fabricative conversions; the fabricative gap is recorded here rather than papered over. The seam to close it later is to persist recovery-supplied values in the report (a v0.2 schema addition), after which offline reconstruction becomes possible.

**Rejected — validate anyway, skipping the un-reconstructable checks.** Emits a green-looking report that silently did not check the fabricated lattice — the report would imply verification that did not happen. **Rejected — read the fabricated values back out of the output file.** Circular: the output is the very thing under test; using it to build the reference guarantees a pass and validates nothing.

## D30 — CLI presentation decisions: exit-code remap, override confidence, ASCII-first rendering (M6)

**Decision.** Three CLI-surface choices (framework is argparse per D6): (a) argparse's default usage-error exit `2` is **remapped to `1`**, because `2` is the spec's *refused* code (§A.2) and the two must not collide. (b) When `inspect --format` overrides the sniffer, the report's `confidence` stays the format's **real sniff score** (with a separate `overridden: true` flag), never a fabricated `1.0`. (c) Renderers are plain-text with `✓/✗/◐/⚠` glyphs only; `--json` emits the pydantic models verbatim with no parallel DTO.

**Rationale.** (a) A CI pipeline keying on exit codes (the whole point of §A.2) would otherwise read an argparse typo as a scientific refusal — a dangerous conflation; one `ArgumentParser.error` override fixes it. (b) Forcing `confidence` to `1.0` on override would hide the fact that the user overrode a poorly-sniffing format — the honest signal is the real score plus an explicit override marker, keeping the report truthful about detection (**P1**). (c) The CLI is a thin presenter (Part 1 §2): rendering must add no information the report lacks, and `--json` must be the report itself so downstream tools never diverge from the API's schema.

**Rejected — let argparse keep exit 2.** Silently overloads the refused code. **Rejected — override ⇒ confidence 1.0.** Fabricates certainty the sniffer did not find. **Rejected — a bespoke JSON shape per command.** A second schema to keep in sync with the pydantic reports, and a drift risk against the future API that returns those same models verbatim (Part 6).

## D31 — Hosted-instance privacy and size posture: 100 MB cap, 30-minute retention, delete-on-download (`MASTER_SPEC` Revision 1.4)

**Decision.** The hosted public instance is a deliberately small, aggressively private demonstration surface; large files are the CLI's and self-hosting's job. Five concrete changes, recorded in `MASTER_SPEC.md` as Revision 1.4 (project owner's directive):

1. **Max upload size: 500 MB → 100 MB** (the software default, not a hosted-only override — the 500 MB figure is removed entirely). `413 FILE_TOO_LARGE` carries `details.self_hosting_url` alongside `limit_bytes`, and the upload page renders the oversize state as a redirect to the CLI install and self-hosting guide (Part 7 §2.2) — a funnel to the free local path, never a dead end.
2. **Stored-byte retention: 7 days → 30 minutes**, uniformly across inputs, persisted Canonical Objects (Part 1 §4.3), and outputs. **Outputs are additionally deleted immediately on the first *completed* download stream**; an aborted stream does not consume the artifact. `410` responses distinguish `details.reason: "expired"` vs `"downloaded"`.
3. **`awaiting_recovery` TTL: 12 hours → 30 minutes**, with a paused job's `expires_at` computed as min(pause + TTL, input `expires_at`) so a paused job can never outlive the bytes it needs to resume. The expiry-to-refusal bright line (Part 4 §3.2) is unchanged.
4. **Enforcement inverts to application-primary.** Bucket lifecycle rules have day granularity and cannot express 30 minutes, so primary enforcement is synchronous delete-on-download plus a minute-cadence retention sweeper (worker-scheduled, driven by `expires_at` in Postgres), with a **1-day bucket lifecycle rule as the platform backstop** and a new fifth alert (retention-sweep lag > 5 min, Part 9 §6.1). The guarantee is stated as two-tier: 30 minutes monitored, 24 hours platform-enforced worst case.
5. **Normative privacy copy** (Part 9 §5.2): *"uploaded files and converted outputs are automatically deleted within 30 minutes — outputs immediately after you download them. File contents are never retained. What persists is the conversion's loss-accounting report (field names, check results, and any recovery parameters you chose) — never coordinates, never file bytes."*

**Field renames (explicit, per the no-silent-rename rule):** `LimitsResponse.retention_days` → `retention_minutes`; `awaiting_recovery_ttl_hours` → `awaiting_recovery_ttl_minutes`. Documentation-only — Part 6 is design intent not yet built (Revision 1.2 status header); renaming now costs nothing, whereas a `retention_days: 0.02` would be a permanently dishonest unit. Propagated to `IMPLEMENTATION_PLAN_v0.5.md` and `_v0.6.md`.

**Rejected — advertising "no data is retained after conversion."** False as stated: reports outlive bytes by design (the audit trail *is* the product's trust claim), and a recovery-supplied lattice or a validation report's measured deviations are data. An overstated privacy claim is exactly the silent dishonesty P1 exists to prevent. The accurate claim is nearly as strong and true. The residual is named precisely in Part 9 §5.2 so a sensitive-data user can judge it — and the zero-residual path (CLI/self-host) is linked from the same copy.

**Rejected — keeping 500 MB as the software default with a hosted-only 100 MB override.** Two numbers to document and test where one suffices; every limit is already an environment variable, so a self-hoster raising the cap is one line of config either way. The owner's directive was to remove 500 MB, and nothing in the corpus depends on it (the chunked-upload deferral in Part 7 §5.2 only gets *easier* at 100 MB).

**Rejected — bucket-lifecycle-only enforcement (the previous stance).** Cannot express a sub-day window; keeping it primary would make the 30-minute promise a fiction enforced only at day granularity. The inversion is forced by the platform; the honest response is to keep the rule as a backstop, monitor the sweeper, and say so — recorded in place in Part 9 §5.2 rather than silently rewritten.

**Rejected — deleting the output on the download *request* rather than stream completion.** Simpler, but a dropped connection would permanently destroy a result the user never received — silent loss of the artifact itself, the worst possible variant of the failure mode this project exists to prevent.

**Rejected — keeping the 12-hour `awaiting_recovery` TTL.** It would either break the 30-minute promise (paused jobs pin their input bytes and canonical object) or strand "resumable" jobs whose bytes are already gone, turning resume into a guaranteed `FILE_EXPIRED`. 30 minutes to answer a recovery prompt is adequate for an interactive session; self-hosters configure longer.

**Considered, deferred — `DELETE /v1/conversions/{conversion_id}` (report/record deletion) for users wanting zero residual.** Additive and compatible, but it weakens the reports-outlive-bytes audit posture for a need the CLI/self-host path already serves completely; revisit only on demonstrated demand (**P6**).

**Consequences acknowledged.** Server-side full re-parse re-validation (Part 5 §4.5 point 2) becomes effectively a same-session operation on the hosted instance; re-thresholding is unaffected (reports only), and the offline CLI path is the durable route — noted in place. The history page's "re-convert while unexpired" window shrinks to 30 minutes on the hosted instance; the expired state was already a first-class rendered state (Part 7 §2.6), so no UI contract changes.

## D32 — Report retention: 30 days on the hosted instance, stated explicitly in the privacy copy (`MASTER_SPEC` Revision 1.5)

**Decision.** Conversion records and their Conversion/Validation Reports are deleted **30 days after the conversion completed** on the hosted instance. Instance-configurable as `report_retention_days` in `LimitsResponse` and the Part 6 §5 limits table; `null` = retain indefinitely (the natural self-host setting for a permanent lab notebook). Enforcement is a daily pass of the existing Part 9 §5.2 retention sweeper over Postgres rows — no bucket rule involved, since reports never touch object storage. Deleting an account deletes its records and reports immediately (cascade), not on the 30-day clock. The normative privacy copy is extended to state this explicitly: *"…and that report is itself deleted after 30 days."* Downstream consequences are recorded in place rather than implied: server-side re-thresholding and the history entry end with the report (Part 6 §2, Part 7 §2.6 — the history row is deleted outright, not shown as a husk); re-thresholding's "always works" claim is scoped to the retention window (Part 5 §4.5, Part 1 §4.3). The durable route after 30 days is the report JSON the user downloaded, which the CLI's offline re-validation consumes with no server state — so the window forecloses nothing.

**Why 30 days.** Long enough to cover every realistic return: re-thresholding under a stricter profile next week, showing a collaborator the loss accounting next month, re-checking an assumption before submission. Short enough that residual metadata (field names, check results, recovery parameters such as a supplied lattice) is bounded rather than accumulating forever against sessions and keys whose owners will never return. It also keeps the whole privacy story statable in one breath: *bytes in 30 minutes, records in 30 days.*

**Rejected — retaining reports indefinitely (the pre-1.5 stance, D31's deferral).** The audit value of a report scoped to an anonymous session or idle API key decays to zero once its owner can no longer return; the retained-data liability does not. Indefinite retention also made the privacy copy's "what persists" clause open-ended — exactly the vagueness P1 disallows in the other direction.

**Rejected — deleting reports with the bytes (30 minutes).** Reduces the product's central trust artifact to a within-session receipt; destroys re-thresholding and history immediately. The report is the point (**P1**).

**Rejected — a shorter window (e.g. 7 days).** Saves nothing measurable (report rows are KB-scale) and cuts off the "come back after the weekend / after review comments" pattern that is the whole reason server-side re-thresholding exists.

**Supersession note.** D31's "Considered, deferred — `DELETE /v1/conversions/{conversion_id}`" is now largely mooted for the hosted instance: automatic 30-day expiry plus account-deletion cascade bounds the residual that endpoint existed to remove. It stays deferred on the same P6 grounds.

## D33 — Hosted-instance platform: one always-free ARM VM running the production compose stack, free-tier object storage, free static hosting (design intent; `MASTER_SPEC` Revision 1.5)

**Decision.** The candidate zero-cost topology for the hosted instance (Part 9 §5.4), recorded now as **design intent** and re-validated at deploy time (roadmap v0.5+), because free tiers churn:

- **Compute — Oracle Cloud Infrastructure "Always Free" Ampere A1 ARM VM** running `docker-compose.prod.yml` verbatim (API + RQ worker + Redis + Postgres). As of mid-2026 the free-tier allowance is 2 OCPUs / 12 GB RAM (halved from 4/24 in June 2026), 200 GB block storage, ~10 TB/month egress — comfortably above the ~2 GB peak-RSS job budget (Part 8 §4) plus API/Postgres/Redis overhead at the hosted instance's 4-concurrent-job cap. This choice **dogfoods the self-hosting story**: the hosted instance *is* a self-hosted instance, so every hosted-deployment fix lands in the artifact self-hosters use (Part 9 §5.4's "primary supported deployment" made literal).
- **Object storage — Cloudflare R2 free tier** (10 GB-month, zero egress fees, S3-compatible — the `06 §5`/`09 §5.2` lifecycle needs only an S3 API). At a 100 MB cap × 30-minute retention, steady-state stored bytes are effectively a rounding error; the same bucket (or Backblaze B2's 10 GB free tier) holds the nightly `pg_dump` for the Part 9 §6.2 backup/restore drill.
- **Static surfaces — GitHub Pages or Cloudflare Pages** for the docs site and the exported Next.js frontend (`07 §5.3` static docs; split-origin layout per Part 9 §5.4), both free with custom domains and TLS.

**Consequence for CI:** container images must be published multi-arch (`linux/arm64` + `linux/amd64`) via buildx — the hosted VM is ARM, self-hosters are mostly x86; a single-arch image would silently fork the two deployments.

**Caveats, recorded honestly rather than discovered at deploy time.** OCI free-tier signup approval is opaque and accounts are occasionally reclaimed or capacity-constrained; idle-reclamation policies exist (a real workload plus the §6.1 monitoring is the mitigation, not a keep-alive hack); and the June 2026 halving of the free allowance is a reminder that any of these tiers can shrink again. None of this is architecture risk — it is deployment risk, which is why the fallback below is a re-deploy, not a redesign.

**Fallback (accepted, not free): a small paid x86 VPS** (e.g. Hetzner-class, ~€4–5/month) running the identical compose file, if OCI proves unreliable in practice. This is the only line item that would make the hosted instance cost money, and it is optional (§5.4: the hosted instance's existence is itself optional).

**Rejected — Render free tier.** Free web services spin down after 15 minutes of inactivity and, decisively, background workers are a paid service type — the RQ worker (the always-on process the Part 9 §4.2 container decision exists for) has nowhere free to run, and a spun-down API breaks the polling contract (`06 §3`).

**Rejected — Railway / Fly.io.** Neither offers a permanent free tier to new accounts as of 2026 (trial credits / pay-as-you-go only); at that point the honest comparison is against the cheaper and simpler VPS fallback, which wins.

**Rejected — serverless assembly.** Already rejected on workload-shape grounds in Part 9 §4.2; nothing about free tiers changes that analysis.

**Rejected — managed-split free stack** (free web host + Neon/Supabase free Postgres + Upstash free Redis + R2). Maximizes the number of third-party suspension policies, connection limits, and cold-start behaviors a zero-revenue demo depends on, still leaves the worker without a free home, and abandons the dogfooding property — four providers to keep honest instead of one compose file.

## D34 — POSCAR scaling factor lives in `provenance.parse_notes`, not `simulation.extra` (post-v0.1 review)

**Decision.** The POSCAR/CONTCAR parser records the source scaling-factor token as a `provenance` parse-note, **not** as `simulation.extra["poscar:scaling_factor"]`. The factor is already folded into the returned lattice vectors (`MASTER_SPEC` Part 3 §4), so it is not independent source information a target must separately carry — it is provenance about *how this file encoded* a lattice ChemBridge now stores in absolute Å.

**Why it had to change (not cosmetic).** `provenance` is deliberately excluded from `field_presence()` (Part 2 §3.11), but `simulation.extra` is a presence-bearing field. No v0.1 exporter can write `simulation.*` (all declare it `NONE`), so pre-flight correctly marked `simulation.extra['poscar:scaling_factor']` as **removed** — yet the POSCAR exporter *always* writes a `1.0` scale line, so the re-parse *always* re-populates that key. The Validation Engine's `absence_conformance` check then saw a "removed" path reappear and failed **every** POSCAR→POSCAR/CONTCAR conversion. Relocating the token to provenance (which carries no ✓/✗ signal) removes it from the pre-flight diff and the completeness invariant entirely, which is the honest classification: a folded-in scale is not losable canonical content.

**Spec reconciliation.** `MASTER_SPEC` §6.1 already states the discriminating rule — the scaling factor belongs in `simulation.extra` "*only if not already reflected in the reconstructed lattice*." Since the parser unconditionally folds it in, the "only if" is never satisfied, so the §8.2 worked example (which showed it in `simulation.extra`) was internally inconsistent with §6.1. Corrected in the same change (`MASTER_SPEC` Revision 1.7): the §8.2 example now carries the factor as a parse-note, and the golden fixtures (`tests/golden/poscar/nacl-primitive/`, `tests/golden/schema/poscar_nacl.json`) and their tests match.

**Rejected — teach the POSCAR exporter capability to carry `simulation.extra`.** The exporter would have to reconstruct the original non-1.0 scale and re-fractionalize, reintroducing exactly the matrix round-trip D8 avoids, to preserve a value that carries no information the lattice does not already hold. **Rejected — suppress the `absence_conformance` check for this one key.** A targeted exception to the check that exists to catch exporter-plan deviations would blunt the very guard the project depends on; the fix is to classify the datum correctly, not to exempt it.
